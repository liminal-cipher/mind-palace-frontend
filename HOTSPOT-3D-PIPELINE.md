# 3D 좌표 계산 · 핫스팟 배치 파이프라인 (memory-walk)

방 GLB(3D 모델)를 1인칭 공간(`memory-walk.html`)에 띄우면, 가구를 인식해 번호 핫스팟(마커)을 찍고, 학습 개념을 그 위에 얹어 "걸어 다니며 외우는" 동선을 만든다. 이 문서는 그 **3D 좌표 계산**과 **핫스팟 배치 원리**를 단계별로, 전문용어는 쉬운 말로 풀어 정리한다.

> 핵심 성질: **데이터 구동(data-driven)** — GLB별 하드코딩이 없다. 같은 규칙이 모든 방·새 GLB에 자동 적용된다. 관련 검증 기록: [GLB-RECOGNITION-VERIFY.md](GLB-RECOGNITION-VERIFY.md).

---

## 0. 용어 빠르게 풀기
- **GLB**: 3D 모델 파일(가구·방이 메쉬 덩어리로 들어있음). 메쉬마다 이름이 있을 수도(`Sofa_01`) 없을 수도(`Object_12`) 있다.
- **메쉬(mesh)**: 삼각형으로 이루어진 3D 형상 한 덩어리(의자 하나, 벽 한 장 등).
- **바운딩 박스(AABB, `THREE.Box3`)**: 어떤 메쉬를 감싸는 **축에 나란한 최소 직육면체**. "이 물건이 차지하는 가로·세로·높이 상자". 물체의 위치·크기를 빠르게 가늠하는 자(尺).
- **앵커(anchor)**: 인식된 가구 1개의 후보 = `{ pos(좌표), label(내부ID), ko(표시명) }`.
- **핫스팟/마커**: 화면에 보이는 번호 구슬. 앵커를 동선 순서로 정렬해 번호를 매긴 것.
- **레이캐스트(raycast)**: 한 점에서 광선을 쏴 무엇에 먼저 맞는지 보는 것. "시선이 가구에 가리나?", "이 (x,z) 바로 아래 바닥 높이는?"를 잴 때 쓴다.
- **정규화(normalize)**: 방마다 실제 크기가 제각각이라, **가로 폭을 7.2로 통일**해 모든 좌표·임계값을 같은 잣대로 다룬다(스캐너 좌표와도 호환).

좌표축 약속: **x·z = 바닥 평면(가로/세로), y = 높이.** 그래서 "마커가 겹쳐 보인다"는 건 **바닥 평면(x,z)에서 가깝다**는 뜻이다(높이 y는 무관).

---

## 1. 정규화 — 모든 방을 같은 자로 (`loader.load` onLoad)
1. 모델을 traverse(전체 메쉬 순회)해 배경(스카이박스 등 `findBackdrops`)을 골라낸다.
2. 배경을 뺀 "실제 방"의 바운딩 박스 `rbox`를 구한다.
3. `scale = 7.2 / max(rbox.가로, rbox.세로)` → 모델 전체를 이 배율로 키우거나 줄여 **가로 폭 7.2**로 통일.
4. 모델을 옮겨 **바닥(rbox.min.y)이 y=0, 중심이 원점**에 오게 한다.

이후 모든 임계값(예: "방 높이의 45%보다 크면")은 정규화된 치수(`roomMax=7.2`, `roomH=size.y*scale`) 기준이라 방 크기와 무관하게 일관된다.

## 2. 가구 검출 — 2단계 (이름표 → 기하학)

### ① 이름표 패스 (`FURN_RE`)
메쉬 이름이 가구 단어(`sofa|chair|table|bed|...`)를 담고 있으면, 그 메쉬의 **바운딩 박스 중심**을 앵커로 추가한다. 이름이 정답이므로 가장 신뢰도 높다.
```
b = Box3(메쉬);  anchors.push({ pos: b.중심, label: 가구단어, ko: furnKO(가구단어) })
```

### ② 기하학 후보 패스 (이름이 없거나 일반적일 때)
이름으로 못 잡는 GLB(`Object_12` 같은)를 위해, **크기·위치·모양**으로 "가구일 법한" 메쉬를 추린다. 후보가 되려면 모든 필터를 통과:

| 필터 | 기준 | 이유 |
|---|---|---|
| 너무 작음 | `maxDim < 7.2×0.03` | 소품·파편 제외 |
| 너무 큼 | `maxDim > 7.2×0.92` | 방 전체 메쉬 제외 |
| 얇고 큰 면 | `thin < 0.028×7.2 && (높이 큼 or 넓음)` | **벽·칸막이** 제외 |
| 납작·거대 | `높이 < 0.012×7.2 && 넓음` | **바닥/천장 슬래브** 제외 |
| 전체 높이 | `높이 > roomH×0.82 && 넓음` | **벽·기둥** 제외(전체높이 가구는 드묾) |
| 천장 부착 | `중심.y > roomH×0.72` | 조명·실링팬이 천장에 안 찍히게 |

> **왜 모양으로 거르나?** 이름 없는 GLB는 벽·바닥도 `Object_N`이라 이름 기반 `STRUCT_RE`로 못 거른다. 그래서 "얇고 크게 선 면 = 벽", "아주 납작·거대 = 바닥" 처럼 **형태**로 구조물을 배제한다(이게 과거 '벽이 수납장으로 과검출'되던 버그의 수정).

통과한 후보는 큰 것부터 정렬(`vol` 내림차순)하고, **생성 시점 거리 솎기(`MINSEP ≈ 0.5m`)**: 이미 있는 앵커와 0.5m 이내면 같은 물체로 보고 버린다.

## 3. 이름 짓기 — `inferFurnLabel` → `geomGuess` → `furnKO`
각 후보의 표시명(ko)은 다음 순서로 결정:
1. **`inferFurnLabel(메쉬)`**: 자기 이름 + **부모·조상 노드 이름 4단계**까지 훑어 가구 단어를 찾는다(3D-FRONT·Sketchfab은 그룹명에 `Sofa_01`이 흔함). 머티리얼명은 오탐이 많아 제외.
2. 실패하면 **`geomGuess(size, 중심높이, roomH)`**: 바운딩 박스 모양만으로 추정. 낮은 것 먼저 → 키 큰 것 → 작은 것:
   - 아주 납작·넓음 → **러그**
   - 바닥 가까이·아주 넓고 낮음 → **침대** / 넓고 낮음 → **소파**
   - 키 크고 깊이 있음(`thin>0.35`) → **수납장**
   - 키 크고 넓지만 앞뒤 얇음(`foot>0.55 && thin<0.5`) → **책장/선반**
   - 키 크고 **양쪽 다 가는 기둥**(`foot<0.45 && thin<0.4`) → **스탠드/조명**
   - 그 외 키 큰 덩어리 → **수납장**
   - 중간높이 평평한 상판 → **테이블** / 앉는 높이 작은 덩어리 → **의자**
   - 나머지 → **소품**
3. **`furnKO`**: 영문 라벨을 한글 표시명으로(예: `dining table`→식탁, `box`→상자).

> ⚠️ **한계(중요)**: geomGuess는 *모양 추측*이라 한계가 있다(예: 사무실 책상이 `식탁`, 넓은 선반이 한때 `스탠드`로 오인). 모양만으론 "책장 vs 수납장 vs 스탠드"를 완벽히 못 가른다. **근본 정확도는 비전(아래 §8)의 몫**이다. geomGuess는 "가구 N" 숫자 대신 자연스러운 이름을 주는 폴백.

## 4. 중복·이상위치 정리 — 솎기 3종 (여기가 자주 문제됨)
마커는 **바닥 평면(x,z)** 에서 인지되므로, 솎기 기준도 바닥 거리여야 자연스럽다.

1. **생성 MINSEP (`≈0.5m`, 3D 거리)** — 후보 추가 시점의 1차 솎기.
2. **`mergeAnchors(list, 1.0)`** — **같은 label**이 1.0m 이내면 병합(수납장 패널 여러 장 → 하나). 단, 기하학 앵커는 label이 `obj0·obj1…`로 전부 고유해 이 단계론 안 합쳐진다.
3. **`declusterAnchors(MIN_SEP≈0.6~1.1m)`** (designRoute 안) — **label 무관 거리 솎기**. 고정물(`FIXTURE_RE`: 창·칠판·수납…)에 가중치를 줘 우선 보존. **이게 실질적 솎기**지만 *편집(나만의 공간) 모드*에선 건너뛴다.
4. **★ XZ 겹침 솎기 (`0.42m`)** — decluster 직후 추가. **바닥물체(러그)+공중물체(벽 수납장·그림)가 같은 바닥 자리에 겹쳐** 중복 마커처럼 보이던 것을 정리. 3D 거리는 높이차 때문에 못 걸렀지만(예: 1.12m), 바닥 거리로 보면 0.27m라 겹침. 0.42m로 좁게 잡아 **카페·교실의 떨어진 의자(≥0.5m)는 보존**.

> 좌표 자체는 **바운딩 박스 중심**을 쓴다. 그래서 러그처럼 납작한 건 중심이 바닥 높이(y≈0)라 마커가 바닥에 붙고, ㄱ자·속빈 메쉬는 중심이 빈 공간일 수 있다 → 이를 솎기·표면부착(§6)으로 완화.

## 5. 동선 설계 — `designRoute` (순서와 선별)
`activeIdx` 없이 입장하면 "공간을 한 바퀴 도는" 순서를 자동 설계한다.
1. **입장 시점 프레임**(`entryFrame`): 더 긴 축의 한쪽 끝(뒤)에 서서 방을 바라보는 좌표/정면/오른쪽 벡터.
2. **클래스별 군집 + 코너 선별**: 같은 종류가 여럿이면 **대각선 4코너 끝**(앞왼/앞오/뒤왼/뒤오로 가장 먼 것)을 골라, 가운데로 쏠리지 않고 방 구석까지 포함.
3. **라벨 보강**: 같은 종류엔 "앞 왼쪽 식탁", "뒤 오른쪽 의자"처럼 위치 수식어를 붙인다.
4. **순서 = 부채꼴 시야 스윕**: 시점에서 좌→우 **각도**로 정렬(머리를 한 방향으로 돌리듯). 각도를 `SECTOR(≈11°)` 버킷으로 양자화해 정렬 비교자가 **추이적**이게 만든다(과거 비추이적 비교자로 엔진별 정렬이 뒤섞이던 버그 수정).
5. 상한 `CAP=max(엔티티수,16)`.

### 노드 수 보강 (합성 앵커)
학습 개념이 검출 가구보다 많으면 부족분을 합성한다. **(구) 빈 바닥 격자**에 좌표를 뿌려 "허공 마커"처럼 보이던 것을 → **(신) 검출된 실제 가구 곁(방 중심 쪽 0.6~0.9m)** 에 배치해 자연스럽게. 가구가 0개인 극단적 경우만 바닥 격자 폴백.

## 6. 마커 좌표 → 화면 위치 (`buildRoute`)
1. `route = ordered.map(a → { pos: a.pos, eye: a.pos.clone(), number: i+1, entity, ... })`
2. **겹친 XZ(0.4m 내) 황금각 나선**: 같은 자리에 둘 이상이면 `황금각(137.5°)`으로 살짝 흩어 마커가 포개지지 않게.
3. **표면 부착(`snapY`)**: 마커 (x,z)에서 **아래로 레이캐스트**해 그 자리 표면 높이 + 18cm에 마커를 얹는다(가구 위에 자연스럽게). 단, 벽 부착물(창문·그림·펜던트)은 아래로 쏘면 바닥에 떨어지므로, 스냅 결과가 원좌표와 1m 이상 차이나면 **원좌표(검출 표면점)** 를 그대로 쓴다.
4. 높이는 `[바닥+0.2, 천장-0.12]`로 클램프.

## 7. 학습 엔티티 매핑 + 카메라
- **엔티티 배정**: ① 저장된 직접 바인딩(`entityId`) 우선(편집·이동에도 유지) → ② 남은 엔티티는 **동선 순서대로**(걷는 순서 = 서사 순서). 과거 타입-친화 배정이 "1번 개념이 9번 지점에" 가던 문제를 폐기.
- **카메라 시점(`clearViewPos`)**: 마커 정면으로 "걸어가" 바라보되, 다른 가구에 **가리면**(레이캐스트 삼각측량, N방향 중 가장 뚫리고 중심을 향하는 곳) 시점을 돌린다(`losClear` 최종 확인).
- **부드러운 이동(`flyTo`+`tick`)**: `다음 →` 누르면 `easeInOutCubic`(가감속) 900ms로 위치·시선을 동시 보간. **트윈 중엔 `walk()`·`clampInsideRoom()`을 멈춰** 카메라를 건드리지 않아 흔들림 없이 활공한다.

## 8. 새 GLB 일반화 · 자가검사 · 정밀 보정
- **무수정 적용**: 데이터 구동이라 새 GLB도 동일 규칙이 자동 적용.
- **자가검사**: 검출 직후 `window.__mpDetectReport = {glb, count, byType, warn}` + 콘솔 `[mp-detect]`. `sparse(<4)`·`skewed(한 종류>5)` 경고로 품질 점검.
- **디버그 덤프**: `window.__mw = { route, anchors, ordered }` — 마커 좌표·라벨·중복을 실측 가능(검증의 *올바른 신호*).
- **정밀 보정(비전)**: 인식이 약한 사물은 **"정밀" 버튼** → 보고 있는 대상 스크린샷을 `gpt-4.1 비전(/api/vision-label)` 에 보내 사람이 즉석 교정. **자동 정확도의 근본 해법은 Azure Vision 검출 부착**(예정).

## 10. 검증 방법(재사용)
```
memory-walk.html?glb=<URL>&palace=public/data/korean_history.palace.json&lmName=test
```
로 로드 후 콘솔에서:
```js
window.__mw.route.map(h => ({ n:h.number, ko:h.ko, x:h.pos.x.toFixed(2), z:h.pos.z.toFixed(2) }))
// XZ 0.42m 이내 쌍(=겹침) 0 이어야, 한 종류 과다(벽 누수) 없어야, 카페·교실 의자 다수는 정상
```
드라이버 페이지에서 숨김 iframe로 여러 GLB를 한 번에 회수해 `byType`·겹침 수를 비교한다(개수만이 아니라 **좌표·중복**을 본다).
