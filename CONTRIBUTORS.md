## 🧠 핵심 의사결정·전환 (왜 지금 모습이 됐나)

처음 구상대로 가지 않고 몇 번 방향을 튼 지점들 — 이 결정들이 지금의 회랑을 만들었다.

### 1) 멘토 피드백 (05-27) — 방향을 잡아준 두 축
- **대표님.** 클러스터링 기반 RAG 배치는 흥미롭지만 **① 클러스터링 후 '정확한 배치'**와 **② 범용 도메인에서의 투명성·신뢰성 검증**이 핵심 보완점. 구현을 넓히기보다 **특정 도메인**(특수교육·박물관 도슨트·관광)으로 좁혀 사용자 시나리오·검증을 구체화하고, 최종은 박물관·관광·기업 실무·디지털 트윈으로 확장 가능한 포트폴리오로. **동선 + speech + RAG 도슨트 투어**가 클러스터링의 강점을 가장 잘 보여주는 형태.
- **책임님.** **Three.js 저사양 최적화** 필수, 옵시디언 등 경쟁 서비스 수준의 사용성. 학습자료 **이미지 처리 부담** → 법률처럼 이미지 적은 도메인도 현실적. **Azure 비용 사전 산정**·Google File Search 등 대안 비교. 데이터 품질·방 나누기 알고리즘 검증 체계. 멀티 에이전트/로컬 LLM **앙상블** 검토. **GraphRAG 먼저 → 필요 부분 커스터마이징**.
- *반영:* 한국사·AI 교안으로 도메인을 좁혀 검증, 책임 있는 AI 6원칙·비용 산정·저사양 모드, GraphRAG 채택 → 모두 이 피드백에서 출발했다.

### 2) 방 구성 구조 (05-28) — 의견 1~5 토론 → **의견5 채택**
무한 확장성 vs 동선 직관성을 놓고 5개 안을 비교했다. (2)집·방·오브젝트는 무한확장 불가, (3·4)오픈공간+시설물 매핑은 시설물이 부족하면 추가 구역 문제. → **의견5**: 1분류=오픈공간 프리셋, 2분류=**건물(최대 개수 제한 → 무한확장 문제 해결)**, 3분류=방/오브젝트(원하는 만큼). 건물 안/밖은 포탈로 이동, 테마별 건물 디자인.

### 3) 원조 풀스택 → GraphRAG 네이티브
초기 설계(시언)는 **DI → GPT-4o Vision → GPT-4o mini → 임베딩 → HDBSCAN/KMeans 방 생성 → UMAP+NetworkX 배치 → HITL**의 직접 조립형이었다. 이후 멘토 제안대로 **GraphRAG**(엔티티·관계·community report)를 본체로 채택하고, 방 경계는 목차로 고정·개념은 위치 기반 배치로 단순·결정화했다.

### 4) 한옥 10개 → 실제 랜드마크 마커 (06-09, 준상)
효석이 한옥 10개를 배치했지만, 준상이 **"도로변 한옥은 부자연스럽고, 10개는 너무 많아 공간-기억 매치가 불안정"**하다고 지적. → **실제 특징적 건물에 GPS 마커**를 다는 게 직관적이고, 로드뷰처럼 익숙한 위치를 마커로 쓰자는 방향으로 전환. "시험 5일 전 학생이 자기 동네·교실을 고른다"는 **교육적 현실 시나리오**로 제품을 다잡았다.

### 5) RAG global 채택 + 정직한 한계 (백엔드)
basic/global/local/drift 중 **global** 채택(비교 질문까지 한국어로 잘 답, 7~8초). 단 **community report에 실린 개념은 강하고, 그래프에서 고립된 개념**(측우기·앙부일구처럼 연결 0)**은 약하다**는 구조적 한계를 직접 재현·기록하고, **근거가 없으면 일반 지식으로 둘러대지 않고 한국어로 솔직히 거절**하게 했다(책임성). 측우기가 첫 방에 묻히던 팰리스 문제와 뿌리가 같다.
